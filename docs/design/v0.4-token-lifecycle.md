# ccbroker v0.4.0 — race-free token lifecycle redesign (v2, post red-team)

Red-team verdict on v1: SHIP WITH FIXES (C-1, C-2 critical; M-1..M-7 mandatory).
This v2 integrates all mandatory fixes. §4 protocol is authoritative.

## 1. Measured facts (2026-07-05/06 incidents + probes)

F1. A refresh (rotation) **immediately revokes the previous access token** (measured:
    acct-A 01:02:05Z rotation → prior gen 401 on profile AND /v1/messages).
F2. A stale refresh token replayed hours after supersession → `invalid_grant` and can
    kill the lineage (broker 22:27Z acct-C). Server semantics consistent with EITHER
    rotating-RT+reuse-revocation OR mostly-stable-RT; design must be safe under both.
F3. One-way `ccb pull` destroyed fresh /login credentials twice (mac SessionStart hook
    00:54:11Z confirmed; B200-2 cron the day before).
F4. Broker served revoked tokens for ~3h because death was only detected at the
    scheduled refresh.

## 2. Claude Code behavior (source-verified + adversarially confirmed, snapshot 2026-04-05)

CC1. Proactive refresh only within **5 min** of expiresAt (client.ts:344-353).
CC2. Before ANY refresh CC re-reads storage; unexpired re-read ⇒ NO refresh, even on
     the force/401 path (auth.ts:1472-82). CC never rotates an unexpired-per-clock token.
CC3. Refresh uses the re-read disk RT under a cross-process lockfile + in-process
     single-flight (auth.ts:1425-45, 1484-1538).
CC4. On 401: re-read storage (cache+keychain cache cleared → bypasses 30s TTL); stored
     token differs from failed ⇒ adopt it, no refresh (auth.ts:1377-88).
CC5. Credential WITHOUT refreshToken: every refresh path is a silent no-op; no crash,
     no logout, no cred deletion; requests error after the retry budget and recover
     when storage updates (verified paths auth.ts:1380-82, 1459-66).
CC6. In-memory cache invalidated on .credentials.json mtime change / keychain 30s TTL
     (auth.ts:1313-36).
CC7. withRetry: 401 → handleOAuth401Error → rebuild client → retry, ≈11 attempts then
     CannotRetryError; staleness inside the budget is absorbed transparently.
CC8. 403 "OAuth token has been revoked" is first-class, treated like 401; CC comments
     attribute it to "another process refreshed the token" (matches F1).
CC9. NO auto-logout: creds are wiped only by explicit /logout, `claude auth logout`,
     or the start of a fresh /login. API failures never clear the store.
CC10. RT rotation is passive client-side: refresh response may omit refresh_token and
     the old one is kept — server may rotate or keep the RT; assume neither.
CC11. Managed/enterprise installs run hourly background polls that can trigger the
     refresh check with no user action (policyLimits, remoteManagedSettings). Not
     active on this fleet; RT-quarantine neutralizes them regardless.

## 3. Root cause of the incident class

Two+ independent refreshers share one OAuth lineage. Any full-credential copy whose
token is expired-per-clock (unmanaged config dir, returning offline machine, another
user's borrowed-account checkout) legally refreshes per CC1/CC2 → revokes the broker
generation (F1) → 30-min pull latency serves corpses (F4) and clobbers newer local
creds (F3) → broker's own later refresh with a superseded RT finishes the family (F2).

## 4. Architecture v2

Invariants:
- I1 (quarantine): refresh tokens live ONLY in the broker store — clients never
  receive one (server strips on serve AND client strips on write). Transient
  exception: a machine that just ran /login holds the new lineage's RT on disk until
  its next sync offers it upstream; that RT must NEVER be destroyed until the broker
  provably holds a live same-account credential.
- I2 (single writer): only the broker calls the refresh endpoint. A fresh /login
  creates a NEW lineage; it enters the system via offer, never via replay.
- I3 (liveness beats timestamps): adoption/overwrite decisions are made on broker-side
  liveness verification, never on client-supplied or cross-clock expiresAt comparison.

### 4.1 Server (ccbrokerd)

S1. **Stripped serving.** GET /v1/creds/{name} returns claudeAiOauth WITHOUT
    refreshToken (keeps accessToken/expiresAt/scopes/subscriptionType/rateLimitTier)
    plus envelope fields {gen, account}. Config `serveRefreshToken` (bool): compat
    escape hatch, used during rollout (M-7), default false in final state.

S2. **Offer/adopt — ACCOUNT-ROUTED** (POST /v1/creds/offer, client bearer).
    The client does NOT pick a cred name; the broker routes by verified account
    (rationale: the 2026-07-06 incident — /login of account B on a machine whose
    @active is A; name-routing would offer B's token to cred A, get email_mismatch,
    then destroy B's RT). Body = full claudeAiOauth WITH refreshToken (else 400).
    Decision procedure (NO expiresAt gate anywhere):
      a. Rate-limit per client (e.g. 6/min) ⇒ 429 (client: keep local + backoff, N-6).
      b. OUTSIDE any cred mutex: verify offered token live — GET /api/oauth/profile
         with the offered accessToken. Non-200 with 401/403 ⇒ adopted=false,
         reason=offer_not_live; 429/5xx/network ⇒ reason=verify_unavailable.
      c. Route across the FULL store (scope gates GET/serving only, NOT offers — any
         client handing in a verified-live credential for a managed account benefits
         the whole fleet; adoption security = liveness + identity + anti-rollback,
         same as in-scope): match by profile account.uuid (primary, A4) then email
         (fallback) → exactly one cred. No match ⇒ reason=unknown_account (an account
         ccb does not manage at all — never overwrite it locally, see C1). Multiple
         match ⇒ reason=ambiguous_account (defensive; migration S8 makes it
         unreachable) — keep local. Records store Account (email) + AccountUUID.
      d. Anti-rollback: if sha256(offered.accessToken) is in that cred's retired-hash
         ring (last 8 retired access tokens; N-8: assumes <8 rotations between the
         slowest machine's syncs — fine for this fleet) ⇒ reject, reason=rollback.
      e. Under the matched cred's mutex: re-check gen unchanged since (b) (else redo
         from (b) once, then reason=conflict); equal token
         (sha256(offered)==sha256(stored)) ⇒ adopted=false, reason=already_current
         (idempotent re-offer). Otherwise ADOPT — liveness verified in (b), identity
         matched in (c), not a rollback per (d); stored health is irrelevant to
         adopting a strictly-verified fresh credential.
      f. On adopt: retire old stored access-hash into the ring, store offered cred
         (full, with RT), bump gen (S3 primitive), set healthState=ok — clears BOTH
         suspect and dead (N-2; the offered token was live-verified seconds ago),
         wake long-pollers, audit `ADOPT name client gen`.
    Response: {adopted, name, reason, gen}. No storedLive here — the freshness proof
    the overwrite gate needs is a property of the cred being WRITTEN, so it lives on
    the GET (below), not on the offer (whose routing fails exactly when the proof is
    needed, e.g. offer_not_live has no verified email to route by).

S2b. **Probed GET (N-1 fix).** GET /v1/creds/{name}?probe=1 — before responding, the
    broker freshly verifies its stored access token via /api/oauth/profile (result
    cached ≤30s per cred to bound probe load; probes never run under the cred mutex).
    Response gains probeLive: true|false. Probe-endpoint-down ⇒ probeLive=false.
    Un-probed GETs are unchanged. Impl: capture the record ONCE and use that same
    snapshot for both the probe and the response body, so probeLive attests exactly
    the token served. Self-heal shortcut: when probe=1 and healthState != ok, run the
    probe BEFORE the 409 decision — a 200 sets healthState=ok and serves normally
    (fluke recovery accelerator).

S3. **Generation + long-poll (C-2, M-4 fixed).**
    - gen is a **persisted strictly-increasing per-cred counter** stored inside the
      record: gen = max(storedGen+1, nowUnixMilli), written atomically with the record
      (temp+rename via existing saveLocked). Survives restart, clock steps, sub-ms
      collisions.
    - Wake mechanism: server-side map[name]*credState{gen int64, wake chan struct{}},
      guarded by ONE mutex; bump = (persist record; lock; state.gen=g; close(wake);
      new wake; unlock). Reader = (lock; capture gen+chan; unlock; compare; wait).
      No missed-wakeup window; long-poll wait holds NO mutex (M-4a/b).
    - GET /v1/creds/{name}?sinceGen=G&waitSec=W (W clamped ≤25, B200-2 uses 15 per
      m-2): return immediately if gen>G, else wait for wake/timeout → 304.
    - credSrv keeps ReadHeaderTimeout only; no ReadTimeout/WriteTimeout that could
      kill an idle long-poll (m-4); per-request write deadline after wake instead.

S4. **Health poller + suspect/dead lifecycle (M-2, M-3 fixed — NO RT replay).**
    - Poller (existing usage/profile 5-min loop) maintains healthState per cred:
      ok / suspect / dead. 401/403-revoked on an unexpired token ⇒ do ONE immediate
      confirm re-probe (fresh profile call, ≥10s later); only a CONFIRMED double-401
      ⇒ suspect: audit `SUSPECT name`, serve 409 to GETs (stops corpse-serving in
      ≤~5min, vs 3h in F4).
    - suspect does NOT refresh immediately (a 401-on-unexpired means our RT is likely
      superseded; replaying it is the F2 event). Recovery paths, in order:
        1. an offer arrives (normal case: /login lands within ~30s via watch) → adopt.
        2. RECLAIM_DELAY (10 min) passes with no offer → ONE last-resort refresh
           attempt (covers the stable-RT/legacy-landmine case): success → audit
           `RECLAIM`, ok; invalid_grant → dead, audit `DEAD_EARLY`.
    - dead is NOT terminal (M-3): poller keeps probing dead creds every 30 min with
      the stored access token; a 200 clears dead (fluke self-heal); offers are always
      accepted per S2 regardless of dead.
    - refreshDue skips suspect/dead creds (their RT must not be replayed by the
      scheduled loop either).

S5. **Refresh loop** unchanged (skew 3600s) for healthy creds; every mutation
    (refresh/adopt/import) bumps gen and wakes long-pollers; every rotation retires
    the previous access-hash into the ring.

S6. **Import** (admin PUT) now: verifies the imported token live via profile and sets
    Account at import time (M-5), enforces Account uniqueness across creds,
    initializes gen per S3, clears suspect/dead. Profile verification unavailable ⇒
    **reject the import** (N-3; operator retries when the endpoint is healthy) — no
    half-initialized Account-empty records can exist, so offer routing always has an
    Account to match.

S8. **Startup migration (A1) — inherited v0.3 store.** On first v0.4 boot:
    - For every record lacking Account/AccountUUID: probe its stored access token via
      profile and populate both. Token dead/probe-fail ⇒ record stays unroutable for
      offers (routing needs identity) until re-imported; WARN in log.
    - Assert Account/AccountUUID uniqueness across records; duplicates ⇒ refuse to
      start with an explicit error naming both records.
    - Until the migration pass completes, offers return reason=migration_pending
      (client: keep local, retry next cycle).
    (adminPut historically dropped Account on re-import — measured live 2026-07-06:
    the acct-B re-import blanked the ACCOUNT column until the next adminRefresh.)

S7. **Health classification (N-4).** The poller and all probes distinguish status
    classes structurally (FetchProfile/FetchUsage return status codes, not collapsed
    error strings): only 401 and 403-revoked count toward suspect/dead; 429/5xx/network
    never affect healthState. (N-5) credState.gen is seeded from the persisted record
    gen at startup and every record mutation goes through the single bump primitive
    (persist → state.gen → close/replace wake) — invariant covered by a dedicated test.

### 4.2 Client (ccb)

C1. **`ccb sync`** (rename of pull; `pull` kept as alias). Per target, per cycle:
      a. read local target.
      b. if local.refreshToken != "" ⇒ POST /v1/creds/offer (ALWAYS — the broker is
         the only adjudicator; account-routed). This is the /login on-ramp.
      c. GET the target's configured cred (@active resolution unchanged), with
         probe=1 iff step (d) will need a liveness proof.
      d. **Overwrite gate (C-1/N-1 final):** write the (client-side stripped) broker
         cred over the local target in exactly these cases:
           - no local RT (plain propagation — nothing to lose), or
           - offer adopted (lineage safely in the broker — even when it was adopted
             into a DIFFERENT cred than @active: local RT is preserved broker-side;
             print notice "adopted <name>; local restored to active <active>; use
             `ccb use <name>` to switch"), or
           - offer definitively rejected as already_current or rollback (proof the
             local token is not a new lineage), or
           - offer rejected offer_not_live AND probed GET returned probeLive==true
             (local cred is dead, broker's is verified-alive NOW — this is the
             legacy-landmine cleanup path).
         NEVER overwrite when:
           - reason=unknown_account (an account the broker does not manage at all —
             deliberate foreign use; keep the local cred, skip this target, and WARN
             loudly + actionably: "account <email> is not broker-managed; its
             credential sits on this disk unmanaged (multi-refresher risk for that
             account) — log out, or import it into the broker"), or
           - reason=ambiguous_account / migration_pending (defensive: keep local), or
           - offer/GET hit verify_unavailable / conflict / 429 (backoff, N-6) /
             404 (old broker) / network error / GET 409 (suspect) / 503, or
             probeLive==false ⇒ keep local, WARN, retry next cycle.
      e. identity sync (.claude.json oauthAccount) unchanged; keychain/file writes
         remain atomic (temp+rename for files; keychain single `security` update) so
         a CC process never reads a torn credential.
    `ccb use` runs the same path. Client NEVER writes a refreshToken to any target
    (strip regardless of broker version).

C2. **`ccb watch`**: foreground daemon: loop { long-poll GET (sinceGen, waitSec) →
    on change OR timeout → full syncCycle }. Local /login creds are therefore offered
    within ≤~waitSec+ε even with no broker-side change. Error backoff 5s→60s jittered.
    Dedicated http.Client (Timeout 90s) separate from the 30s default client.

C3. **`ccb setup`**: daemon is the DEFAULT install (M-6): launchd KeepAlive (mac),
    systemd-user service Restart=always + linger note (bw2/WSL), nohup + cron
    `*/5 ensure-alive` wrapper where no systemd (B200-2). ALWAYS also installs the
    30-min cron/timer `ccb sync` as watchdog fallback (m-3) and prints the real
    outage bound if the daemon dies (≤ interval per rotation). Cron liveness also
    refreshes status.json so statusline `~stale` reflects a dead watcher.

C4. Plugin SessionStart hook → `ccb sync`; plugin 0.4.0.

### 4.3 Race/failure analysis v2

R1 dual-refresher: clients hold no RT ⇒ only the broker refreshes. /login machines
   hold a NEW lineage's RT briefly; they offer it, never refresh it (CC won't: token
   fresh for 8h; sync strips it within seconds-to-minutes). ✓
R2 /login clobber (F3): overwrite gate requires adopted, a definitive
   already_current/rollback rejection, or offer_not_live + fresh probeLive proof;
   every failure mode of the offer path (profile outage, old broker 404, 429,
   conflict, clock skew — timestamps aren't consulted at all) leaves the local RT
   intact. Worst case: a genuinely-dead legacy RT survives until a cycle where the
   broker's stored cred probes live ⇒ then cleaned up. ✓
R3 rotation blip (F1): rotation → gen bump → long-poll wake → all watch daemons write
   within ~1-2s; CC sessions recover via CC4 (forced keychain re-read bypasses the
   30s TTL) inside the ~11-attempt retry budget. Cron-only/dead-daemon machine bound:
   ≤ cron interval, stated honestly and mitigated by C3 defaults. ✓ (m-1 documented)
R4 legacy landmine rotation: confirmed-401 → suspect in ≤~5min, 409 to clients (no
   corpse-serving); no immediate RT replay (F2 avoided); /login offer recovers in
   ~30s when the user acts, or one delayed last-resort reclaim after 10 min covers
   the stable-RT case. A reclaimed lineage can no longer out-compete a later /login:
   adoption ignores expiresAt entirely (I3). ✓
R5 offer races: single cred mutex for commit + optimistic re-check around the
   out-of-mutex profile verify; rate limit bounds churn; anti-rollback ring blocks
   replay/pinning; equal-token offers are idempotent no-ops. Two concurrent /logins:
   last adopt wins and retires the loser's access-hash; the loser machine's next
   sync re-offers its now-retired token ⇒ definitive rollback rejection ⇒ overwrite
   ⇒ converges (or, if its token was also revoked, offer_not_live + probeLive). ✓
R6 broker down / GET 503 / suspect 409: client keeps local, CC5 degrades gracefully,
   no client-side refresh exists to corrupt anything. ✓
R7 clock skew/steps anywhere: no decision consumes cross-clock expiresAt (I3); gen is
   a persisted counter, not wall-clock (C-2). ✓

## 5. Rollout (M-7 order)

1. Implement + release v0.4.0.
2. Deploy broker to CT111 with `serveRefreshToken: TRUE` (full compat: old clients
   keep self-heal; nothing strips yet).
3. Upgrade all 4 clients to v0.4.0 + convert schedulers to watch-daemon (C3);
   verify: long-poll latency, offer path (real /login), C-1 gate (reject test),
   suspect flow on a canary (acct-B controlled external rotation).
   N-7: during this window, do NOT /login on a not-yet-upgraded machine (its old
   one-way pull still clobbers fresh logins); keep the window short.
4. Flip broker `serveRefreshToken: false` + `ccb sync` everywhere once → fleet
   disks lose RTs → quarantine active.
5. Update memory + README migration section.

## 6. Test matrix (red-team gaps folded in)

Unit:
- strip: server serve + client write never emit refreshToken (incl. serveRefreshToken
  toggle); envelope gen/account present.
- gen: monotonic under backward clock step; persists across store reload; distinct
  under same-ms double mutation.
- offer decision table (account-routed): not-live / verify-unavailable /
  unknown_account / rollback-hash / equal-token(already_current) / adopt into
  non-active cred / happy-path adopt; adopt clears suspect AND dead (N-2) and the
  same-cycle GET no longer 409s; rate limit → 429 → client keep-local+backoff (N-6);
  concurrent offer-vs-refresh under -race.
- probed GET: probeLive true/false; probe cache ≤30s; probe-endpoint-down ⇒ false;
  never under cred mutex.
- health classification: 429/5xx never suspect (N-4); gen seeded from store on
  restart, long-poll resumes with pre-restart sinceGen (N-5).
- import: rejected when profile unavailable (N-3); Account uniqueness enforced.
- migration (A1): boot with legacy records (Account empty) → populated via probe;
  dead-token record stays unroutable + WARN; duplicate Account refuses start;
  offers during migration → migration_pending → client keeps local.
- routing (A3/A4): uuid-primary email-fallback; offer from a client whose scope
  excludes the matched cred still adopts (store-wide routing) while that client's
  GET of it stays 403; unknown_account keeps local + warns; ambiguous_account
  defensive path.
- probed GET: probe self-heal (200 on suspect → ok → serves); record snapshot used
  for probe and body is identical.
- overwrite gate: adopted / no-local-RT / storedLive / ALL failure branches keep local
  (esp. profile-500 and 404-old-broker) — the C-1 regression test.
- suspect lifecycle: single fluke 401 does NOT brick (confirm-probe passes → ok);
  double-401 → suspect → 409; offer during suspect adopts; RECLAIM_DELAY last-resort
  refresh success and invalid_grant→dead paths; dead self-heals on 200 probe.
- long-poll: wake on bump, 304 on timeout, no missed wake between capture and wait
  (-race), refresh proceeds while long-poll in flight and offer verifying (M-4).
- sync decision + use path; watch backoff/reconnect; setup daemon unit rendering.
Integration (httptest fake upstream implementing BOTH semantics models — rotating-RT
with reuse-revocation AND stable-RT with access-revocation-on-rotate):
- rotation → 2 fake watch clients receive new gen ≤ poll latency.
- /login flow: local full cred → offer → adopt → local stripped next cycle.
- external rotation → suspect → offer recovery; and → delayed reclaim under stable-RT.
- old-ccb + new-broker(serveRefreshToken=true) behavioral compat; new-ccb + old-broker
  404 path keeps local RT.
